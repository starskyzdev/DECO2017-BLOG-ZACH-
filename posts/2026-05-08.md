Based on what we learned in class is that every visible UI component should be map directly to structured data. After finalizing our low-fidelity wireframes, the next logical step was building a Data Definition Document (DDD). This document translates our interface needs into a tangible database schema, bridging the gap between user experience design and backend development.

Our schema is built around four core tables, each serving a specific UX goal.

## 1. The `users` Table

![My diagram](assets/users.png)

Provided by the BlaBla Corp template, this table stores our authenticated users. Because the Mojo.js session cookie automatically exposes the `user_id` and `username` to all routes, we do not need to build complex authentication logic ourselves. We simply need to link our community content back to these user IDs to ensure ownership and accountability.

## 2. The `bakes` Table

![My diagram](assets/bakes.png)

This is the central table of our application. It stores the main bake records. Everything else in the database either belongs to a user or belongs to a bake.
The columns cover process variables (`hydration`, `flour_mix`, `room_temp`, `bake_temp`), qualitative data (`notes`, `result`, `question`), and an `is_shared` boolean.

The `is_shared` column is crucial for our UX. It defaults to `0` (private). This directly addresses our research finding that bakers want a private space to log their experiments. When a user actively chooses to share their bake for feedback, the value flips to `1`, pushing the record to the public community feed.

## 3. The `comments` Table

![My diagram](assets/comments.png)

This table stores the community critique. Alongside the standard `content` text and foreign keys, we added two specific columns: `feedback_type` and `confidence`.

**A Deliberate Trade-Off:** We currently store feedback types as comma-separated text values (e.g., "crumb,oven_spring") rather than creating a fully normalized, separate join table. While strict normalization is best practice for large-scale production, using a simple TEXT column keeps our prototype's implementation manageable while still delivering the exact structured feedback experience we want for the user.

## 4. The `starter_logs` Table

![My diagram](assets/starter.png)

This table powers the daily habit loop requested in the brief. It stores one row per feeding event, tracking the `fed_at` timestamps, `rise_percent`, and qualitative `notes`. By keeping this table lightweight, we make daily tracking frictionless.

## Data-Driven Accessibility

![My diagram](assets/all.png)

When mapping our UI to data, we also locked in our accessibility decisions. Sourdough baking is a hands-on, messy process. Users often have flour on their hands and divided attention.

To support this real-world context, we planned for large interaction targets and clear, descriptive labels on all forms rather than relying on disappearing placeholder text. This ensures our application easily meets the AA accessibility compliance standards required by the brief, while actively solving a genuine usability problem for our bakers.
